# 요구사항 설계

반복 질문으로 요구사항을 구체화한 뒤, 문서를 생성합니다.

## 1단계: 반복 질문

`$ARGUMENTS`에 대해 빈칸과 약한 부분을 찾아 **한 번에 하나씩** 질문합니다.

### 질문 규칙

- **한 번에 질문 하나.** 절대 두 개 이상 질문하지 마세요.
- `AskUserQuestion`을 사용하고, 가능하면 2-4개 구체적 선택지를 제시하세요.
- 이전 답변을 기반으로 다음 질문을 만드세요.

### 질문 차원

필요한 순서대로 질문합니다. 모든 차원을 다 물어볼 필요는 없습니다.

1. **범위** -- 1차 출시에 포함할 것과 뺄 것
2. **엣지 케이스** -- 예외 상황에서 시스템이 어떻게 반응하는지
3. **성공 기준 구체화** -- 모호한 기준을 입력/출력으로 변환
4. **전제 확인** -- 암묵적으로 가정한 기술/환경 결정
5. **도메인 고유 리스크** -- 해당 도메인에서 흔히 놓치는 함정. 일반적인 확인 질문이 아니라 도메인 전문가만 떠올릴 수 있는 질문을 하세요.
   - 예: 결제 → "부분 환불 시 쿠폰 할인분은 어떻게 처리하나요?"
   - 예: 댓글 → "대댓글이 달린 댓글을 삭제하면 대댓글은 어떻게 되나요?"
   - 예: 장바구니 → "품절된 상품이 담긴 상태에서 주문 시도하면?"

### 종료 조건

다음 중 하나면 질문을 멈추고 2단계로 이동합니다:

- 사용자가 "됐어", "충분해", "저장해" 같은 신호를 보냄
- 주요 빈칸이 모두 채워짐
- 3회 이상 질문한 후 새로운 발견이 없음

## 2단계: 문서 생성

질문 결과를 반영하여 요구사항 문서를 작성하세요. 템플릿은 `references/template.md`를 읽고 그 형식에 맞춰 작성하세요.

### 작성 규칙

- **구현 방법은 쓰지 마세요.** "무엇을"에만 집중하세요. 구현 팀이 기술 스택을 자유롭게 선택할 수 있도록 기술 중립적이어야 합니다.
- 모호한 부분은 추측하지 말고 미결정 사항에 나열하세요.
- 성공 기준은 "이 입력이면 이 출력" 형식으로 작성하세요.
- 범위 제외 항목에는 **왜 빼는지** 이유를 함께 쓰세요.

### 문서 완성 체크리스트 (저장 전 검증)

문서를 저장하기 전에 다음을 확인하세요:

- [ ] 모든 성공 기준이 구체적 숫자/문자열로 입력 → 출력을 명시하는가
- [ ] 구현 기술(라이브러리, 언어, DB, 저장소)이 포함되지 않았는가
- [ ] 범위 제외 항목에 "왜 빼는지" 이유가 있는가
- [ ] 미결정 사항이 추측 대신 명시적으로 분리되었는가

## 3단계: 저장

최종 문서를 `artifacts/` 디렉토리에 저장합니다.

- 파일명: `artifacts/<feature-name>/spec.md`
- 저장 전 사용자에게 파일명 확인

## 4단계: spec.yaml 추출

spec.md 저장 후, 시나리오를 `artifacts/spec.yaml`에 구조화하여 추출합니다.

### 추출 규칙

1. `references/spec-schema.yaml`을 읽어 형식을 참조한다
2. spec.md의 각 시나리오/성공 기준에서 구조화된 시나리오를 추출한다
3. ID는 `{FEATURE}-{NNN}` 형식 (피처명 대문자, 예: `KANBAN-001`)
4. 기존 `artifacts/spec.yaml`이 있으면 scenarios에 append한다
5. 없으면 `version: 1`로 새로 생성한다
6. examples의 input은 사용자가 넣는 값, expect는 화면에서 확인 가능한 상태만 쓴다
7. 사용자에게 추출 결과를 보여주고 확인 후 저장한다
